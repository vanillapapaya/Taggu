# Taggu 공개·배포 전략 메모

> 다른 세션에서 이어서 논의하기 위한 핸드오프 문서. "이 프로젝트를 어디에, 왜, 어떤 신원으로 공개할까"를 정리.

## 맥락 (다음 세션이 알아야 할 것)

- 만든 사람: **현업 데이터 분석가**, 소프트웨어 개발은 비전공·입문. AI 코딩 도구 도움 많이 받아 완성.
- 프로젝트: **Taggu** — 100% 로컬에서 도는 한국어 짤·이미지 자동 태깅 & 의미 검색 앱. (WD14 캐릭터 인식 + VLM 한국어 태그/설명 + CLIP 유사도. 로컬 GPU 또는 클라우드 API 선택.)
- 목적 두 갈래: (1) **취업 포트폴리오** (2) **실사용자·피드백·재미**.
- 본인 고민: "전문가도 아닌데 공개하기 부끄럽다", "DC에 올리는 게 이상한가", "신원 특정이 걱정", "DC에서 고용될 가능성?"

---

## 공개하면 얻는 강점 (핵심 질문 답)

### 1. "만들었다 → 남이 봤다"로 격이 올라감
혼자 만든 토이는 흔하다. **실제로 공개되어 누군가 써본 흔적**(다운로드 수, 이슈, 피드백 반영 커밋)이 붙으면 "끝까지 완성하고 책임진 사람"이 된다. 면접에서 한 줄이 강해짐: *"공개 후 N명이 사용, 피드백 받아 X 개선."*

### 2. 데이터 분석가 채용에서 차별화
분석가 풀에서 "동작하는 AI 앱을 직접 배포해본 사람"은 드물다. LLM/AI 활용 능력 + 문제정의→측정→결정 사고를 *결과물로* 증명. 이력서 한 줄 + 링크로 끝.

### 3. 피드백 = 다음 개선 항목 = 또 다른 이야깃거리
실사용자 요청("이 기능 없냐", "이거 안 됨")이 로드맵이 된다. 이 반복 자체가 포트폴리오의 깊이가 됨.

### 4. 검색 유입·평판 (장기)
GitHub 공개 + 블로그 글이 검색에 걸리면, 채용 담당자가 *능동적으로 발견*하는 경로가 생김. 시간이 지날수록 복리.

### 5. 본인 효능감
"부끄럽다"의 해독제는 완성도가 아니라 *남에게 보여본 경험*. 한 번 공개하면 다음이 쉬워짐.

## 어디에 올릴까 — 채널별 비교

| 채널 | 주는 것 | 신원 노출 | 채용 연결 | 추천도 |
|---|---|---|---|---|
| **GitHub 공개 repo** | 코드 증빙, 링크 허브 | 실명/핸들 | 직접(지원 시 확인) | 필수 |
| **긱뉴스 (news.hada.io)** | 양질의 기술 노출 | 선택 | 높음 (채용자가 봄) | 강력 추천 |
| **링크드인 / 커리어리** | 분석가 리크루터 접촉 | 실명 | 가장 높음 | 분석가면 필수 |
| **개발 블로그 글** | 검색 유입, 사고력 증명 | 선택 | 중장기 | 추천 |
| **DCInside (개발/AI/서브컬 갤)** | 실사용자, 거친 피드백, 재미 | 익명 권장 | 거의 없음 | 재미·유저용만 |

핵심: **채용 노출은 긱뉴스·GitHub·링크드인. DC는 채용 경로가 아니라 사용자·피드백·재미 경로.** 둘을 섞지 말 것.

---

## DC에 올리는 것에 대한 결론

- **이상하지 않다.** 짤 호더 + 한국어 + 애니 편향이라 DC가 오히려 최적 사용자층. 포트폴리오상 "한계"인 애니 편향이 여기선 강점.
- **단, 채용 기대는 하지 말 것.** DC는 익명 구조라 누가 감탄해도 당신에게 닿을 수 없고, 채용자가 인재 찾으러 오는 곳이 아님.
- **"DC 좋게 안 본다"는 stigma**는 *고용주가 내 실명 검색 → DC 발견* 경로에서만 문제. 포트폴리오(실명)와 DC(익명)를 **절대 서로 링크하지 않으면** 자동으로는 안 이어짐.

## 신원 분리에 대한 솔직한 정리

- **완벽한 익명은 보장 불가.** 작정한 적대자에겐 문체·핸들 재사용·업로드 시간·EXIF·프로젝트 고유성으로 연결될 수 있음.
- **하지만 현실 위협은 "전문 추적자"가 아니라 "내가 무심코 흘리는 단서".** 이건 통제 가능.
- 노출 단서 체크리스트 (DC에 익명으로 올릴 경우):
  - [ ] GitHub repo의 실명/이메일 (커밋 author, LICENSE) 정리 또는 비공개
  - [ ] 양쪽에서 같은 닉네임 재사용 금지
  - [ ] 업로드 이미지/스크린샷 EXIF·경로(`C:\Users\실명...`) 노출 확인
  - [ ] DC 글에 GitHub 실명 링크 직접 게시 금지 (EXE만 배포하거나 별도 공개 계정)
- **추천 절충:** GitHub repo는 공개해두되 DC엔 직접 링크 안 함 → "남이 발견하게" 두면 신원 노출 최소 + 반응은 봄.

## 한 줄 결론

> 취업이 목적이면 **GitHub 공개 + 긱뉴스 + 링크드인**이 정답. DC는 채용이 아니라 사용자·피드백·재미용이며, 하더라도 신원만 의식적으로 분리하면 됨. 공개 자체의 강점은 "완성도"가 아니라 **"남이 써봤다는 증거 + 분석가 차별화 + 효능감"**.
