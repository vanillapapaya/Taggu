# 인프라 및 배포

## 호스팅 구조

```
[Cloudflare DNS]
    │
    ├──→ yoink.app → [Vercel] (Next.js 프론트엔드)
    │
    ├──→ api.yoink.app → [Railway] (FastAPI 백엔드)
    │
    └──→ r2.yoink.app → [Cloudflare R2] (이미지 CDN)

[Railway 내부]
    ├── FastAPI 서버 (Web)
    ├── Celery Worker (이미지 처리, 벡터화)
    ├── PostgreSQL
    ├── Redis
    └── Qdrant (Docker)
```

## 배포 전략

### 프론트엔드 (Vercel)
```
GitHub Push → Vercel 자동 빌드 → Preview 배포 (PR) / Production 배포 (main)

환경변수:
- NEXT_PUBLIC_API_URL
- NEXT_PUBLIC_R2_URL
- NEXT_PUBLIC_GA_ID (Google Analytics)
```

### 백엔드 (Railway)
```
GitHub Push → Railway 자동 빌드 → 배포

Dockerfile:
- Python 3.11
- FastAPI + Uvicorn
- CLIP 모델 (빌드 시 다운로드, 캐싱)
- e5-large 모델 (빌드 시 다운로드, 캐싱)

환경변수:
- DATABASE_URL
- REDIS_URL
- QDRANT_URL
- TWITTER_BEARER_TOKEN
- CLOUDFLARE_R2_ACCESS_KEY
- CLOUDFLARE_R2_SECRET_KEY
- CLOUDFLARE_R2_BUCKET
- JWT_SECRET
- ANTHROPIC_API_KEY (캡셔닝용, 선택)
```

### ML 모델 로딩
```
CLIP (ViT-B/32): 약 350MB
e5-large: 약 1.3GB

전략:
- Railway 빌드 시 모델 다운로드 후 Docker 이미지에 포함
- 또는 모델을 R2에 저장하고 서버 시작 시 다운로드 + 캐싱
- Cold start 시간: 모델 포함 시 30초, 다운로드 시 2-3분
```

## 모니터링

### 핵심 지표
| 지표 | 도구 | 알림 기준 |
|------|------|----------|
| API 응답 시간 | Railway 내장 | p95 > 2초 |
| 에러율 | Sentry | 5분간 에러 10건 이상 |
| DAU/MAU | Google Analytics | - |
| 검색 쿼리 | PostgreSQL 로그 | - |
| 피드 세션 시간 | 커스텀 이벤트 | - |
| R2 스토리지 | Cloudflare 대시보드 | 80% 이상 |

### 로깅
```
- 검색 쿼리 + 결과 수 + 응답 시간 → 검색 품질 개선
- 피드 스와이프 패턴 (체류 시간, 좋아요) → 추천 개선
- 에러 로그 → Sentry
```
