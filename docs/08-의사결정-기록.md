# 08 — 의사결정 기록

주요 기술/디자인 선택의 근거. 미래의 자신에게 "왜 이걸 골랐지?" 답하는 문서.

## 아키텍처

### Why Vanilla JS + 단일 HTML 파일

**선택지:** React/Vue + 빌드 파이프라인 / Vanilla / Svelte / HTMX
**선택:** Vanilla, 단일 `templates/index.html`

**근거:**
- 단일 사용자 도구라 빌드 파이프라인 ROI 낮음
- Ctrl+F5 한 번에 즉시 반영 — 개발 사이클 매우 빠름
- 의존성 0 (npm/yarn/vite 없음) — 새 머신에서 즉시 동작
- 파일 하나에서 검색하면 모든 코드 보임 — LLM 컨텍스트 친화
- 3000줄이라 monolithic 부담은 있지만 라우팅/상태가 단순해서 견딜 만

**언제 바꿀까:** UI가 별도 페이지(설정 페이지 등 모달 외) 추가될 때, 또는 컴포넌트 재사용성이 정말 필요할 때.

### Why FastAPI

**선택지:** Flask / FastAPI / Django / Litestar
**선택:** FastAPI

**근거:**
- async 자연스러움 (인덱싱 background thread + HTTP 동시)
- Pydantic으로 request 검증 자동
- 자동 OpenAPI 문서 (개발 시 디버깅 편함)
- Uvicorn 단독 실행 가능, 별도 nginx/gunicorn 불필요
- Python 생태계 표준 → ML 라이브러리와 자연스럽게 결합

### Why SQLite (vs Postgres)

**선택지:** SQLite / Postgres / MongoDB
**선택:** SQLite WAL 모드

**근거:**
- 단일 사용자, 단일 머신 → 별도 DB 서버 거대 오버킬
- 단일 파일 → 백업/이동/복원 단순
- WAL 모드로 동시 read/write 충분
- 사용자가 DB Browser for SQLite 같은 도구로 직접 열어볼 수도 있음
- ALTER TABLE 마이그레이션이 단순 (_MIGRATIONS 리스트 한 줄)

**한계:** 10만 row+ 가벼운 분석 쿼리 가능. 그 이상 시 sqlite-vec / FAISS 보강.

## AI / 모델 선택

### Why WD14 EVA02-Large

**선택지:** WD14 (vit/swinv2/eva02 등 여러 변종) / DeepDanbooru / illustration2vec
**선택:** WD14 EVA02-Large (2024-07 최신 버전)

**근거:**
- SmilingWolf 최신 (v4 미존재)
- EVA02-Large가 정확도 최상
- ONNX 형식으로 CPU 추론 가능 (GPU 없는 사용자도)
- Danbooru 5000+ 캐릭터 학습
- 한국어는 직접 안 다루지만 character_aliases.json으로 매핑

**한계:** 2024년 중반 이후 신규 캐릭터 못 잡음 → 그 자리를 CCIP가 메움.

### Why CCIP (캐릭터 학습)

**선택지 (연구 후):** CoOp / Tip-Adapter / NCM / CCIP / 직접 fine-tune
**선택:** CCIP 임베딩 + NCM(centroid) + cosine 검색

**근거 (연구 결과):**
- CCIP는 deepghs가 애니 캐릭터 동일성 전용으로 학습한 ONNX 모델
- WD14가 모르는 신규 캐릭터도 "동일 인물 여부"는 잘 잡음
- ONNX → CPU 추론 가능 (120ms/장)
- centroid 학습 = online mean, 신규 클래스 추가 cost 0 (행 1개 INSERT)
- Tip-Adapter / CoOp는 정확도 비슷한데 신규 클래스 추가 시 재학습 부담 큼
- WiSE-FT는 full FT 대상이라 우리 시나리오엔 오버킬

**근거 (학술):**
- arXiv:2207.09519 (Tip-Adapter) — training-free k-NN이 ImageNet 16-shot 65.5%
- FSCIL VLM 서베이 — incremental setting에서 단순 prototype + cosine 강함
- AniWho (arXiv:2208.11012) — 애니 캐릭터 prototypical net 89% (5-way 5-shot)

**임계값 (CCIP cluster meta):**
- OPTICS eps = 0.18 (cosine distance) → cosine sim ≥ 0.82 = 동일 캐릭터
- 추천 노출 임계값 0.70 (보수적, 사용자 판단 유도)
- 자동 부착 임계값은 향후 0.85+로 검토

### Why Multi-Provider VLM 추상화

**선택지:** Local Qwen만 / OpenAI만 / 추상화
**선택:** Local + 3개 API의 4-provider 추상화

**근거:**
- 사용자 GPU 보유 여부가 결정적 진입 장벽 (~95% GPU 없음)
- API 모드 추가 = 잠재 사용자 풀 20배 확대
- Provider별 가격/품질 트레이드오프 (Gemini Flash 거의 무료 vs OpenAI 비쌈)
- Provider abstraction은 코드 ~300줄, ROI 매우 좋음
- 동일 사용자라도 케이스별로 Local(품질) / API(속도)을 골라 쓸 수 있음

### Why bitsandbytes 4bit (vs autoawq)

**선택지:** autoawq / GPTQ / bitsandbytes
**선택:** bitsandbytes nf4 + double quant

**근거:**
- autoawq는 Windows + Python 3.13 wheel 없음 (triton 의존성도 Linux 전용)
- bitsandbytes는 Windows + Python 3.13 wheel 존재
- 실측 결과: Qwen2.5-VL-7B FP16(16GB) → bnb 4bit(5.51GB), 로딩 45s, 추론 4.9s
- 품질 손실 거의 없음 (nf4 + double quant)

## UX 결정

### Why Edge --app 모드 (vs Electron / pywebview / Tauri)

**선택지:** Electron / pywebview / Tauri / Edge --app
**선택:** Edge / Chrome `--app` + 별도 user-data-dir

**근거:**
- Electron 부담 (~150MB 번들)
- pywebview는 WebView2 의존 + 파일 드래그 권한 제한
- Tauri는 Rust + 추가 빌드 체인
- Edge --app은 OS에 이미 있음 → 추가 다운로드 0
- 진짜 데스크톱 앱 같은 UX (URL 막대 없음, 즐겨찾기와 격리)
- 단점: 사용자가 Edge나 Chrome 설치 필요 (Windows 11엔 기본 포함)

### Why 헤더 단일 줄 강제 (flex-wrap: nowrap)

**선택지:** wrap (다중 줄) / nowrap (단일 줄) / 햄버거 메뉴
**선택:** nowrap + 검색박스만 elastic + 헤더 액션 별도 줄 X

**근거 (사용자 요청):**
> "맨 윗줄에 나오는 것들은 무조건 한 줄에 나오게 해줘"
- 사용자가 강하게 선호. 와이드 데스크톱 환경 가정
- 좁은 창에선 검색 입력만 짧아짐 (search-box: flex 1 1 0; min-width: 0)

### Why 분홍색 검색 버튼

**선택지:** 노랑(accent) / 분홍(char) / 보라(user) / 회색
**선택:** 분홍 (--char #FF8899)

**근거 (사용자 요청):**
- 사용자가 직접 지정
- 노란색은 [랜덤]이 이미 차지 → 시각 구분 필요
- 분홍이 캐릭터 칩과 같은 톤이라 디자인 시스템과 일관

### Why 이모지 제거 정책

**선택지:** 이모지 사용 / 기능 심볼만 / 텍스트만
**선택:** 헤더 monochrome 심볼(⋯⚙?⌨ℹ★)만, 장식 이모지(🤖🎉📂 등) 전부 제거

**근거 (사용자 강 피드백):**
> "이모지를 보면 두드러기가 올라와"
- 가이드 11스텝, README 등에서 장식 이모지 모두 제거
- ❓(U+2753, emoji 블록) → ?(ASCII) 같은 식으로 잠재 emoji-rendering도 제거
- monochrome Unicode 심볼(★ ⌨ 등)은 기능적이라 허용

## 데이터 / 상태

### Why 사용자 라벨 자동 보존 (relocalize 사용자 편집 보존)

**계기 (사고):**
- 초기 `relocalize`는 `wd_chars_ko`를 무조건 alias 매핑 결과로 덮어쓰기
- 사용자가 수동으로 추가했던 200개 캐릭터 칩 손실
- 복구 거의 불가능 (WAL checkpoint, file history 미사용, OneDrive 밖)

**수정:**
- alias 매핑 결과 + 매핑으로 안 만들어지는 기존 한국어 항목 = 사용자 것 → 합침
- 위험 작업 직전 자동 DB 백업 도입 (`db_backups/`)
- 향후 같은 류 사고 시 즉시 복구 가능

**교훈:** 사용자 수동 편집물은 1급 신호. 자동 작업이 절대 덮어쓰면 안 됨.

### Why centroid 학습 시 추천 무시는 학습 X

**선택지:** "추천을 사용자가 무시했다 = false" 신호로 학습 / 무시는 학습 안 함
**선택:** 사용자가 추천 칩을 명시적으로 클릭한 경우에만 centroid 강화

**근거:**
- 사용자가 추천을 무시한 이유는 다양 (다른 작업 중, 못 봄, 실수)
- false negative로 학습하면 noise만 쌓임
- "명시적 신호만 학습"이 안전한 디폴트 (conservative bias)

## 도구 / 워크플로우

### Why uv (vs pip / poetry)

**선택지:** pip / poetry / uv / rye
**선택:** uv

**근거:**
- 매우 빠름 (Rust 구현)
- 표준 requirements.txt 호환
- venv 생성/관리 단순
- pyproject.toml 불필요 (단순 도구)

### Why PyInstaller onedir (vs onefile)

**선택지:** onefile / onedir
**선택:** onedir

**근거:**
- onefile은 매 실행마다 임시 폴더에 압축 해제 → 느림 (몇 초 추가)
- onedir은 zip만 풀면 끝, 실행 즉시
- 사용자가 `_internal/`을 보더라도 OK (이상한 경험 X)
- 빌드도 빠름

### Why 단일 git repo (vs separate frontend/backend)

**선택지:** mono / multi-repo
**선택:** mono

**근거:**
- 단일 사용자 1인 프로젝트
- 프론트엔드도 backend 안에 있는 단일 HTML (별도 리포 의미 X)
- 버전/배포 동기화 단순

## 기능 범위 / 통합

### Why 유튜브 짤 생성을 Taggu에 격리 모듈로 통합

**선택지:** 별도 스크립트로 분리 / Taggu에 깊게 통합 / Taggu에 격리 모듈로 통합
**선택:** 격리 모듈 — `ytgif.py`를 Taggu 안에 두되, 의존성·실패를 한 구석에 가둠

**근거:**
- Taggu의 실제 정체성은 "이미지 태거"가 아니라 **"내 짤 허브"** — 짤을 만들고·모으고·검색하는 한 워크플로. 짤의 상당수가 유튜브에서 딴 GIF/캡처라, 생성은 본체가 다루는 콘텐츠의 공급원.
- 만들기 → `유튜브짤/`에 저장 → 기존 `index_folder` 증분 인덱싱이 mtime 비교로 **새 한 장만** 처리 → 검색 자동 편입. 통합 면적: 엔드포인트 2개 + 모달 1개 + 기존 파이프라인 재사용.
- 공개 전략(→ [[10-공개-전략-메모]]) 관점에서도 "완전 로컬 짤 허브" 스토리가 "또 하나의 태거"보다 강함.

**자동 설치 + 런타임 격리 (둘 다 유지):**
- 자동 설치: `yt-dlp`(영상 구간 다운로드)와 `imageio-ffmpeg`(ffmpeg 바이너리를 venv 안에 동봉)를 `requirements.txt`에 포함 → `uv pip install -r requirements.txt` 한 번으로 venv에 자동 설치. winget/uv-tool 수동 단계 불필요.
- 런타임 격리는 그대로: ffmpeg/yt-dlp는 subprocess로만 호출, 경로 탐색은 PATH → `~/.local/bin` → imageio-ffmpeg 동봉본 순. 없으면 `/api/youtube/tools`가 알려주고 **그 버튼만 비활성** + 설치 안내. 검색·태깅 본체는 무영향.
- 처음엔 "fragile해서 requirements에서 빼자"였으나(아래 교훈), 사용자가 자동 설치 편의를 우선 → 포함으로 전환. 대신 graceful degradation을 남겨 "빠져도 본체 무영향"이라는 격리의 핵심은 보존.

**트레이드오프:** imageio-ffmpeg가 ffmpeg 바이너리(수십 MB)를 끌고 오므로 기본 설치/EXE 용량이 늘어남. yt-dlp는 유튜브 변동 시 `uv pip install -U yt-dlp`로 갱신 필요.

**언제 바꿀까:** 용량/유지보수 부담이 과해지면 별도 `requirements-youtube.txt`로 분리하거나 기능 토글로 뺀다.

## 메타 / 협업

### Why 사람-시간 추정 안 함

**선택지:** "이거 0.5일", "1주 작업" / 컴포넌트 수, 위험도로만 표현
**선택:** 후자 (사용자 명시 요청)

**근거:**
> "어차피 니가 바로바로 해주는데 몇일 걸린다는게 의미가 있니"
- LLM이 작업 수행 → 인간 발 시간 추정 무의미
- 대신: 건드릴 컴포넌트 수 / 위험도 / 반복 가능성 / 결과물 크기로 표현

### Why 메모리 시스템 적극 사용

**근거:**
- 사용자의 선호(이모지/시간추정 등)는 한 번 들으면 기억하는 게 효율
- 향후 동일 실수 방지 + 매번 다시 묻는 마찰 제거
